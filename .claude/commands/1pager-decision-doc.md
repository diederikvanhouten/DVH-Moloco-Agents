# ✏️ /skill-1pager-decision-doc

> **Source:** Moloco Chief of Staff operating system — exported 2026-May-09
> **Scope:** Generic methodology — adapt tool names, org structure, and personas to your context.

**Purpose:** Create decision documents, strategy 1-pagers, and alignment docs that drive a decision or gain alignment.
**Trigger:** Any request for a 1-pager, decision doc, strategy memo, or alignment document.

These are distinct from briefs and deep dives — they are authored documents meant to drive a decision or gain alignment, not to summarize information.

---

## Format Rules — Hard Constraints
- **No section headers.** Do not use H1, H2, or any heading-level formatting. The document is pure prose with inline bold only.
- **No tables.** Tables signal a status report, not a decision document. Use prose with inline structure instead.
- **No bullet lists.** If options must be numbered, number them inline (1. / 2. / 3.) within a paragraph — not as a formatted list.
- **No colored callouts, boxes, or visual containers.** These are presentation-layer elements, not document elements.
- **1-pager discipline.** The default length is one page of dense prose. Err toward compression, not completeness.

---

## Structure of a Good Decision Document

**1. Situation paragraph — tell the story first.** Open with how we got here: what decision was made previously, what was built or learned since, and why the context has now changed. Do not open with "The decision is made" or a statement of the conclusion. The reader needs the narrative arc before the ask.

**2. What this document is for.** One sentence stating the purpose: a smooth transition, a resource decision, a timeline, etc. Not a conclusion — a frame.

**3. Decisions to be made.** State them explicitly and numbered. Use bold inline labels: **Decision #1:** and **Decision #2:**. Not as a header — inline.

**4. Options per decision.** Number them inline (1. / 2. / 3.) within the body paragraph. Frame each option by the value it delivers or the risk it avoids — not by who suggested it. Recommended option gets "(recommended)" inline, not a separate call-out.

**5. Closing question in italic** when the document is meant to drive alignment rather than land a final recommendation. Use a direct, open-ended question that frames what the reader needs to resolve.

---

## Tone Rules
- Declarative and direct — but not abrupt. The situation paragraph should read like a clear-eyed summary of context, not a legal finding.
- Frame options by their structural value, not by their origin or who advocated for them.
- Dependency framing applies: for each option, make explicit what breaks if it isn't chosen or what it enables if it is.
- Do not hedge. If there is a recommended option, say so plainly.

---

## Writing Style
**Prose:** Short, dense paragraphs (3–5 sentences max). Declarative voice: "We will...", "The goal is...", "[Company] will..." — never hedging language.

**Quantification:** Always specific. Metrics drive every claim. If there is no number, attach a directional target.

**Vocabulary:** Domain-specific throughout. Zero generic language: never "drive value," "improve efficiency," or "enhance experience."

**Dependency framing:** Always paired — *what do you need?* → *what breaks if you don't get it?*

---

## Deliverable Format — Hard Rule
**Your primary notes/docs system is the deliverable.** Write the document there and confirm by surfacing the URL as the response deliverable. Do not produce a downloadable file unless explicitly requested.

---

## Hard Guardrails
- This skill overrides any default formatting behavior. No headers. No bullets. No tables. No exceptions.
- The situation paragraph is always first — never open with the conclusion or the ask
- Options are framed by value, not by origin or advocate
- Default length = one page. Never ask if they want it longer — compress until asked to expand
- **Write-back to your docs system is mandatory** — completing the document without the write-back is a hard failure.

---

## Known Failure Modes
- Adding H2 headers to separate sections — everything is inline bold within prose paragraphs
- Opening with "The goal of this document is..." instead of the narrative situation
- Using bullet points for options instead of inline numbered prose
- Attributing options to the people who suggested them instead of framing by structural value
- Exceeding one page without compression attempt first
- **Producing a downloadable file when the deliverable is a doc** — the notes/docs system is always the primary output unless explicitly asked for a file. Producing both is wrong. Producing only a file is wrong.
